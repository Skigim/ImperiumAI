```md
## Development-Time Studio Roster

The repository uses a development-time two-tier operating model.

### Tier 1: Domain Consultants

- `Economy Analyst` owns economic intent, formulas, reserve thresholds, hauling math, and recovery criteria.
- `Combat Tactician` owns defensive and offensive combat intent, target priority, survivability thresholds, and safe mode criteria.
- `Expansion Planner` owns scouting requirements, room scoring, reserve or claim policy, and bootstrap milestones.
- `Industry Analyst` owns terminals, labs, boosts, factory chains, market behavior, and multi-room resource balancing intent.

Tier 1 roles define what correct behavior is. They do not write production code.

### Tier 2: Engineering Core

- `Lead Architect` owns intake, scope control, architecture alignment, CPU-risk classification, and Tier 1 conflict resolution.
- `State & Memory Engineer` owns interfaces, memory schema, cache strategy, serialization rules, and cleanup contracts.
- `Implementation Dev` owns production code changes and must keep implementation bounded, DRY, and architecture-consistent.

Tier 2 roles define how approved behavior is represented and executed. They do not silently change Tier 1 gameplay semantics.
```

- [ ] **Step 2: Add the operating rules section below after the new roster section**

```md
## Operating Rules

- Standard work moves through this artifact chain: `domain brief` -> `architecture impact note` -> `state contract` -> `implementation change set` -> `verification report`.
- Fast-track is allowed only for config-only changes that mutate existing approved tuning surfaces.
- Fast-track is not allowed when a change adds memory fields, adds runtime control flow, changes ownership boundaries, or touches kernel scheduling, cross-room coordination, or shared contracts.
- Fast-track verification must still include an automated TypeScript compile or lint pass.
- Any nontrivial high-CPU or cross-room feature must declare a degraded mode before implementation.
- Any memory addition must declare owner, lifetime, read path, write path, cleanup path, and expected cost class.

### Global Priority Hierarchy

The Lead Architect resolves conflicts using this default order:

1. Kernel safety
2. Emergency defense
3. Room survival
4. Memory integrity and state safety
5. Economy continuity
6. Expansion progress
7. Industry optimization
8. Opportunistic offense
```

```md
These templates operationalize the Project Imperium development-time roster.

- `fast-track-checklist.md`: Required qualification and safety checks for config-only balance changes.

Do not overwrite the templates themselves. Copy the template, then save the completed artifact to the destination path listed above.

The canonical design source for this workflow is [../specs/2026-04-11-agent-roster-operating-model-design.md](../specs/2026-04-11-agent-roster-operating-model-design.md).
```
